# Aurora v1.0 开发记忆与决策记录

## 1. 开发时间线（按版本）
- 2026-05-22：v0.1 完成最小闭环。
- 2026-05-23：v0.2 完成多模型对比（LSTM / LogReg / MLP）。
- 2026-05-23 ~ 2026-05-24：v0.3 完成稳健性增强（rolling + threshold + error analysis）。
- 2026-05-24：v0.4 完成 pandas 与 SQL 风格特征工程迁移一致性验证。
- 2026-05-24：v1.0 完成交付冻结与文档化。

## 2. 关键技术决策
1. 先“课程可交付”，再谈业务集成。
- 原因：课程目标是完整 ML 工程闭环，不是生产系统重构。

2. 以 MLP 作为主线模型。
- 原因：在当前数据和任务下，v0.2 对比中 MLP 指标更稳定、可解释成本更低。

3. 引入 rolling 验证与阈值调优。
- 原因：单次切分偶然性较强，rolling 更能说明稳定性；阈值可以按业务偏好调节误报/漏报。

4. 做 pandas -> SQL 风格特征迁移对比。
- 原因：为未来对接数据平台/策略引擎做准备，先验证“迁移不改结果”。

## 3. 踩坑与修复
1. 文档乱码风险
- 现象：旧文档在部分环境显示乱码。
- 修复：v1.0 全部文档统一 UTF-8 重写，旧文档归档到 `docs/archive/`。

2. 临时目录与缓存污染
- 现象：`__pycache__`、临时 sweep 目录混杂在主路径。
- 修复：统一归档到 `outputs/v1/_legacy_cleanup`，主目录保持干净。

3. 版本输出散落
- 现象：早期输出在 `outputs/` 根目录。
- 修复：归档到 `outputs/v1`，形成按版本管理。

## 4. 当前稳定事实（可用于答辩）
- v3 final（`outputs/v3/final/metrics.json`）
  - f1=0.5882, precision=0.5200, recall=0.6771, accuracy=0.5260, roc_auc=0.5474
- v4 compare（`outputs/v4/compare_metrics.csv`）

## 5. 对后续工作的指导
- 若目标是课程答辩：优先保证“复现、解释、对比”三件事清晰。
- 若目标是系统集成：先做数据契约与接口隔离，再做自动化调度与监控。
