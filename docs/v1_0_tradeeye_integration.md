# Aurora v1.0 对接 TradeEye 方案（文档版）

## 1. 对接目标
把 Aurora 的“次日上涨概率 + 风险特征”作为 TradeEye 的一个可消费信号源，支持：
- 研究态：结果展示与解释
- 生产态：定时运行、告警推送、策略联动

## 2. 对接边界
- Aurora 负责：数据处理、特征构建、模型训练/推理、指标产出。
- TradeEye 负责：任务调度、通知编排、跨策略聚合、前端展示。

## 3. 最小数据契约（建议）
建议 Aurora 输出统一结构（CSV/JSON 均可）：
- `trade_date`: 交易日
- `ts_code`: 股票代码
- `up_prob`: 次日上涨概率
- `y_pred`: 二分类结果
- `threshold`: 当前阈值
- `model_version`: 模型版本（如 v1.0）
- `feature_engine`: pandas/sql
- `run_id`: 本次运行唯一标识

## 4. 三阶段接入路线
### 阶段 A：离线文件接入（最快）
- Aurora 运行后输出 `predictions.csv`。
- TradeEye 在本地或 CI 读取该文件并拼接展示。
- 适合先验证“接口和解释口径”。

### 阶段 B：脚本接口接入
- Aurora 提供 `export_tradeeye_payload` 脚本，输出标准 JSON。
- TradeEye 调用该脚本并消费结果。
- 适合稳定批处理流程。

### 阶段 C：服务化接入
- Aurora 暴露轻量 API（如 `/predict`）。
- TradeEye 按任务/策略实时拉取。
- 适合高频和多策略联动。

## 5. 关键技术点
1. 特征一致性
- 统一 `feature_engine` 和特征列定义，避免线上线下漂移。

2. 阈值治理
- 在 Aurora 侧保存阈值来源（rolling_median / val_best），TradeEye 展示时同步透出。

3. 可追溯性
- 每次输出必须带 `run_id`、配置摘要、关键指标。
- 发生异常时可定位到具体配置与数据快照。

4. 失败保护
- 没有新数据时保留上次结果并打标。
- 推送前检查关键字段完整性（日期、代码、概率、版本）。

## 6. 验收标准（建议）
- 接口层：TradeEye 成功读取 Aurora 标准输出并展示。
- 结果层：展示值与 Aurora 原始产物完全一致。
- 稳定层：连续 5 个交易日任务成功，无空推送、无字段缺失。
- 可解释层：每次推送可回溯到 run_id、阈值来源、模型版本。

## 7. 实施清单（可执行）
1. 固化数据契约字段。
2. 在 Aurora 增加“对接输出”脚本（文件/JSON）。
3. 在 TradeEye 增加读取与校验逻辑。
4. 在调度层加入失败重试与告警。
5. 完成联调记录与验收报告。
